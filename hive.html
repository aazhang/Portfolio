Mobile Design

Hive

Hive is a project management and team communication tool. It brings tasks, messages, and files into an all-purpose dashboard. In late 2016, Hive set out to redesign the mobile experience. You can download the app here.

My Role

I led product design for this redesign. I gathered user feedback, wireframed, prototyped, delivered high-fidelity mockups, and worked closely with our team of engineers during development.

The Problem

The Hive mobile app is built with Cordova, which essentially packages our current web app as a mobile app. The previous design reflected this and it was time for a tailored mobile experience. Although we plan to build a native app in the future, for this redesign we set out to build a MVP mobile experience with the main goals of improving usability and performance.

Understanding the User

I began this project by interviewing users to understand user needs, behaviors and pain points of a mobile app counterpoints. Research revealed how the app fits into the lives of the users. For example, users often use the mobile app when they are on the go or when they first wake up.

Discovering that the context in which a user used the mobile app greatly differed from a desktop app allowed us to prioritize what features to focus on for version 1. We set out to launch an MVP of the desktop service with focus on three main scenarios:

1. I want to quickly communicate with my team members.

2. I want to see what’s on my plate for today.

3. I want to stay up to date on changes to projects and actions.

Prototyping Interactions

Since nearly everything on our desktop app can be dragged and dropped, I spent a lot of time thinking of how interactions would work on mobile and created early prototypes to share with the team. I wanted the interactions to help the user maintain contect and improve percieved performance, however I was cautious that it didn’t feel disruptive either.

In the end many of these interactions did not make it into V1 launch as we found that it’s hard to get it right through Cordova. We have put interactions on the roadmap for when we launch a native app.

Following Best Practices

For V1 of the redesign, I frequently referred to iOS design guidelines and set out to ensure that the mobile app followed best practices such as typography, styling and size of navigation bar / tab bar, and larger touchpoints.

From Stories to Features

Our decision to focus on three main uses cases instead of a full-featured mobile application was very intentional. As an early-stage startup, our product is always in flux and we are releasing new features daily. This strategy allowed us to refine the core experience and not waste time redesigning features that might be nixed in a few weeks.

"I want to quickly communicate with my team members"

Research showed that the core use of the mobile app was to communicate with team members. With this insight, we brought messaging to the forefront. I opted to separate the navigation panel from the chat panel to allow users to easily scan through chat groups and team members. I also added a top section so that all unread messages are above the fold.

"I want to see what’s on my plate for today"

The second tab is for actions with “My Actions” as the default. This tab only contains actions assigned to the user, sorted by priority. This way a user can navigate to the second tab and see what tasks he/she need to work on for that day.

"I want to stay up to date on changes to projects and actions"

With new actions, comments, changes to statuses, and more there can be a LOT of notifications in Hive. To ensure the notifications page would actually be useful I grouped updates by action and then chronologically. I also separated updates from mentions as testing with the team revealed that users are often looking for notifications specifically relating to them.
